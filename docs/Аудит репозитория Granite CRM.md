# Аудит репозитория Granite CRM

**Дата аудита:** 3 апреля 2026 г.
**Репозиторий:** [aipunkfacility/granite-crm](https://github.com/aipunkfacility/granite-crm)
**Аудитор:** Manus AI

## 1. Введение

Был проведен комплексный аудит репозитория Granite CRM. Проект представляет собой локальную CRM-систему, предназначенную для личного использования на локальном сервере, без выхода в интернет и без других пользователей. Основные функции включают управление контактами, распределение по областям и массовую email-рассылку. Бэкенд реализован на Python с использованием FastAPI, а фронтенд — на ванильном JavaScript, HTML и CSS с использованием IndexedDB (Dexie.js) для локального хранения данных.

В ходе аудита были проанализированы архитектура, безопасность, качество кода и общая структура проекта. С учетом личного и локального характера использования, акцент сделан на надежности, предотвращении потери данных, удобстве использования и поддержке кода. Ниже представлены подробные результаты и рекомендации по улучшению.

## 2. Архитектура и структура проекта

Проект имеет классическую клиент-серверную архитектуру, где сервер выступает в роли API для работы с файловой системой (хранение JSON-файлов базы данных) и отправки email-сообщений, а клиентская часть представляет собой Single Page Application (SPA), которое кэширует данные в IndexedDB.

### Положительные аспекты

-   **Разделение ответственности:** Серверная часть логично разделена на роутеры (`db.py`, `send.py`, `backup.py`, `template.py`) и сервисы (`email.py`, `batch.py`, `backup.py`).
-   **Асинхронность:** Использование FastAPI и `asyncio` позволяет эффективно обрабатывать запросы и выполнять фоновые задачи (например, массовую рассылку писем).
-   **Локальное хранение:** Использование IndexedDB на клиенте позволяет приложению работать быстро и снижает нагрузку на сервер, что идеально для личного использования.
-   **Система бэкапов:** Реализовано автоматическое создание резервных копий при изменении или удалении файлов базы данных, что критически важно для предотвращения потери данных.

### Области для улучшения

-   **Отсутствие базы данных на сервере:** Хранение данных в виде отдельных JSON-файлов (`db/*.json`) подходит для небольших объемов данных, но при росте базы это может привести к проблемам с производительностью и целостностью данных. Для личного использования это может быть приемлемо, но стоит учитывать потенциальные ограничения.
-   **Синхронизация состояния:** Текущая реализация синхронизации между клиентом (IndexedDB) и сервером (JSON-файлы) может привести к конфликтам при одновременной работе нескольких вкладок браузера, что может вызвать потерю данных или несогласованность.
-   **Глобальное состояние:** В `batch.py` используется глобальный словарь `jobs` для хранения состояния рассылок. При перезапуске сервера все данные о текущих рассылках будут потеряны. Для личного использования это означает, что незавершенные рассылки придется запускать заново после перезапуска сервера.

## 3. Аудит безопасности

Поскольку система предназначена исключительно для личного использования на локальном сервере и не имеет внешнего доступа, многие традиционные угрозы безопасности (например, внешние атаки, несанкционированный доступ других пользователей) значительно снижаются. Однако остаются аспекты, важные для целостности данных и предотвращения случайных проблем.

### Аспекты, требующие внимания (с учетом локального использования)

-   **Раздача статики:** Сервер раздает всю корневую директорию проекта как статические файлы (`app.mount("/", StaticFiles(directory=BASE_DIR, html=True), name="root")`). Это означает, что файлы конфигурации (`config.json`), логи (`logs/crm_server.log`) и бэкапы могут быть доступны по прямым ссылкам через локальный веб-сервер. Хотя это не является критической уязвимостью для *внешнего* злоумышленника, это может привести к случайному раскрытию чувствительной информации или ее изменению другими локальными процессами или пользователями на той же машине.
-   **CORS:** В `server/__init__.py` настроен максимально разрешающий CORS (`allow_origins=["*"], allow_credentials=True`). Для личного локального использования это не представляет прямой угрозы безопасности, но является плохой практикой, которая может быть унаследована, если проект когда-либо будет развернут в более открытой среде. В локальной среде это может позволить другим локальным веб-приложениям взаимодействовать с CRM без ограничений.
-   **XSS (Cross-Site Scripting):** В клиентском коде (например, `render.js`, `email-sender.js`) HTML-разметка формируется путем конкатенации строк. Хотя используется функция `esc()` для экранирования данных, риск XSS остается, особенно при вставке данных в атрибуты (например, `onclick`). Для личного использования это может привести к неожиданному поведению интерфейса или потере данных из-за некорректно отображаемых элементов.
-   **Path Traversal:** В роутерах `db.py` и `backup.py` используется `os.path.basename(filename)` для защиты от выхода за пределы директории. Это хорошая практика, которая помогает предотвратить случайное изменение или удаление системных файлов, даже в локальной среде.
-   **Отсутствие аутентификации и авторизации:** Для личной CRM, работающей только на локальном сервере, отсутствие аутентификации не является критической проблемой, если машина защищена. Однако, если к машине имеют доступ другие пользователи, или если сервер будет доступен в локальной сети, это может стать проблемой. В данном контексте это скорее вопрос удобства и контроля доступа, чем критической уязвимости.

## 4. Качество кода и потенциальные баги

### Серверная часть (Python)

-   **Валидация данных:** В `models.py` используются базовые модели Pydantic без строгой валидации (например, не проверяется формат email-адресов). В `db.py` валидация JSON-данных ограничивается проверкой типа (dict/list) и размера. Это может привести к сохранению некорректных данных, что усложнит их дальнейшую обработку.
-   **Обработка ошибок:** В большинстве случаев ошибки перехватываются общим блоком `except Exception as e` и возвращаются клиенту с кодом 500. Это может затруднить отладку и понимание причин сбоев.
-   **Управление зависимостями:** В `requirements.txt` указаны только основные зависимости. Рекомендуется использовать инструменты вроде `pip-tools` или `poetry` для фиксации версий всех транзитивных зависимостей, чтобы обеспечить воспроизводимость окружения.

### Клиентская часть (JavaScript)

-   **Глобальные переменные:** Использование глобальных объектов (`Render`, `State`, `EmailSender`) может привести к конфликтам имен и усложняет тестирование и дальнейшую разработку.
-   **Hardcoded URL:** В `email-sender.js` жестко задан `SERVER_URL: 'http://localhost:8000'`. Это делает невозможным использование системы на другом порту или хосте без изменения кода, что снижает гибкость.
-   **Управление DOM:** Массовое использование `innerHTML` для обновления интерфейса неэффективно и может приводить к потере состояния элементов (например, фокуса ввода), а также к потенциальным проблемам с производительностью на больших объемах данных.

## 5. Рекомендации по улучшению

### Надежность и предотвращение потери данных

1.  **Ограничение раздачи статики:** Настройте `StaticFiles` так, чтобы раздавались только необходимые для работы фронтенда директории (`css`, `js`, `index.html`), а не весь корень проекта. Это поможет избежать случайного доступа к файлам конфигурации, логам или бэкапам.
2.  **Синхронизация состояния:** Если планируется работа с CRM в нескольких вкладках или браузерах, рассмотрите механизмы для более надежной синхронизации данных между клиентом и сервером, чтобы избежать конфликтов и потери данных.
3.  **Персистентность задач:** Для фоновых задач (email-рассылки) используйте очередь задач с персистентным хранилищем (например, Celery + Redis/SQLite) вместо хранения состояния в памяти. Это позволит продолжить рассылку после перезапуска сервера.

### Удобство использования и поддержка

1.  **Улучшенная валидация данных:** Добавьте строгую валидацию данных с помощью Pydantic на сервере (например, `EmailStr` для email-адресов). Это поможет поддерживать чистоту данных и предотвратить ошибки.
2.  **Конфигурация фронтенда:** Вынесите `SERVER_URL` в конфигурационный файл или определяйте его динамически на основе `window.location.origin`. Это упростит развертывание на другом порту или хосте.
3.  **Рефакторинг фронтенда:** Рассмотрите возможность использования современных легковесных фреймворков (например, Vue.js, React или Alpine.js) для управления состоянием и рендеринга интерфейса. Это значительно упростит разработку, поддержку и улучшит пользовательский опыт, а также решит проблемы с `innerHTML` и XSS.

## 6. Заключение

Granite CRM представляет собой функциональный прототип локальной CRM-системы, который успешно решает задачу управления контактами и email-рассылок для личного использования. С учетом того, что система предназначена для одного пользователя на локальном сервере, основные риски безопасности, связанные с внешними атаками, минимальны. Однако для обеспечения максимальной надежности, предотвращения потери данных и удобства дальнейшей разработки и использования, рекомендуется обратить внимание на улучшение механизмов хранения данных, синхронизации и качества клиентского кода. Предложенные рекомендации помогут сделать вашу личную CRM еще более стабильной и приятной в работе.
