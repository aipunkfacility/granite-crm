# Granite CRM: Руководство по email-рассылкам

## Назначение документа

Технический справочник по рассылкам в Granite CRM. Содержит архитектурные решения,
известные проблемы и ограничения. Обновляется при изменениях в подсистеме рассылок.

---

## 1. Типы шаблонов

| body_type | Описание | Как хранится в БД | Как рендерится |
|-----------|----------|-------------------|----------------|
| `plain` | Plain text | `CrmTemplateRow.body` — текст с плейсхолдерами | Оборачивается в `<pre>` с `html.escape()`, передаётся как `MIMEText("plain")` + `MIMEText("html")` |
| `html` | Готовый HTML | `CrmTemplateRow.body` — полный HTML-документ | Передаётся как `MIMEText("html")` напрямую, plain-альтернатива генерируется автоматически через `_html_to_plain_text()` |

Один шаблон = один формат. Если нужен и plain, и HTML — создаются два отдельных шаблона.

### Валидация

- `body_type="html"` допускается **только** при `channel="email"`. Telegram и WhatsApp не поддерживают HTML.
- `body_type="plain"` допускается при любом канале (`email`, `tg`, `wa`).
- Бэкенд возвращает `400 Bad Request` при попытке создать HTML-шаблон для `channel="tg"` или `channel="wa"`.

## 2. Плейсхолдеры

### Доступные плейсхолдеры

| Плейсхолдер | Описание | Пример значения |
|-------------|----------|----------------|
| `{city}` | Город компании | Москва |
| `{company_name}` | Название компании | Гранит-М |
| `{website}` | Сайт компании | granit-m.ru |
| `{from_name}` | Имя отправителя | Александр |
| `{contact_name}` | Имя контакта | Иван |
| `{phone}` | Телефон компании | 74951234567 |

### Правила использования плейсхолдеров

1. **Плейсхолдеры допустимы только в текстовом содержимом тегов.** Примеры:

   ```html
   <!-- ✅ Правильно: плейсхолдер внутри текста тега -->
   <p>Здравствуйте, {company_name}!</p>
   <td>Город: {city}</td>

   <!-- ❌ Неправильно: плейсхолдер в атрибуте тега -->
   <a href="https://example.com/{company_name}">Ссылка</a>
   <img alt="{company_name} логотип" src="...">
   ```

2. При `body_type="html"` значения плейсхолдеров экранируются через `html.escape()` перед подстановкой. Это предотвращает XSS: если `company_name` содержит `<script>`, он будет вставлен как текст, а не исполнен.

3. Незаполненные плейсхолдеры логируются как warning. Шаблон отправляется с пустыми значениями на месте незаполненных плейсхолдеров — письмо не блокируется.

### Текущий статус персонализации

На текущем этапе HTML-шаблоны **не используют плейсхолдеры** — письмо отправляется
как есть, без подстановки. Персонализация будет добавлена позже. Архитектура
поддерживает плейсхолдеры из коробки: достаточно добавить `{company_name}` и т.д.
в HTML-код шаблона — рендеринг сработает автоматически.

## 3. Подготовка HTML-шаблона

### Обязательные шаги перед загрузкой в CRM

1. **Удалить скрытый plain-text `<div>`.** Некоторые HTML-шаблоны содержат блок вида:

   ```html
   <div style="display: none; white-space: pre-wrap; ...">
     PLAIN TEXT VERSION HERE
   </div>
   ```

   Этот блок **не работает** как plain-текст альтернатива. Почтовые клиенты,
   показывающие только text/plain, увидят весь HTML-код, а не этот div.
   Plain-версия генерируется автоматически через `_html_to_plain_text()`
   (BeautifulSoup: вырезает `<script>`/`<style>`, извлекает текст с переносами строк).

   **Действие**: удалить этот div из HTML-файла перед загрузкой в CRM.

2. **Проверить внешние изображения.** Текущий шаблон использует:

   ```html
   <img src="https://memorial-img.netlify.app/male.jpg" ...>
   ```

   **Риски**:
   - Gmail/Outlook блокируют внешние изображения при первом открытии (пользователь видит «Показать картинки»)
   - Хостинг (Netlify) может быть недоступен или URL может измениться
   - Без загрузки картинки письмо показывает пустое место вместо примера работ

   **Действие**: рассмотреть удаление внешних изображений из шаблона или
   встраивание через CID (attachment) в будущей версии. На текущем этапе —
   оставить как есть, но зафиксировать риск.

3. **Убедиться в наличии `</body>`**. Tracking pixel (1x1 PNG для отслеживания
   открытий) вставляется перед `</body>`. Если тег отсутствует — pixel добавляется
   в конец HTML. Но лучше всегда иметь `</body>`.

4. **Inline-стили обязательно.** Почтовые клиенты вырезают `<style>` в `<head>`.
   Все стили должны быть в атрибуте `style=""` каждого тега.

5. **Табличная вёрстка.** `<div>` + CSS flex/grid не работают в Outlook.
   Использовать `<table>` для лейаута.

## 4. Механика отправки

### Путь письма от шаблона до получателя

```
CrmTemplateRow (body_type, body, channel)
       │
       ▼
  template.render(**kwargs)
       │  Для plain: str.replace() без экранирования
       │  Для html:  str.replace() + html.escape() на значениях
       ▼
  rendered (готовый текст/HTML с подставленными значениями)
       │
       ▼
  EmailSender.send()
       │  Для plain: body_html = <pre>escaped(body_text)</pre> + pixel
       │  Для html:  body_html = rendered + pixel перед </body>
       │             body_text = _html_to_plain_text(rendered)
       ▼
  MIMEMultipart("alternative")
       ├── MIMEText(body_text, "plain")   ← для клиентов без HTML
       └── MIMEText(body_html, "html")    ← для современных клиентов
       │
       ▼
  SMTP → получатель
       │
       ▼
  CrmEmailLogRow (tracking_id, status, sent_at)
  CrmTouchRow (channel="email", direction="outgoing")
  CrmContactRow (email_sent_count++, last_email_sent_at, funnel_stage)
```

### Tracking pixel

Каждое письмо содержит 1x1 прозрачный PNG с уникальным URL:

```
https://<BASE_URL>/api/v1/track/open/<tracking_id>.png
```

Когда почтовый клиент загружает изображение — фиксируется `opened_at` в `CrmEmailLogRow`.

**Ограничение**: Gmail и Outlook могут предварительно загружать изображения через
прокси-сервер, что приводит к ложным срабатываниям. Это известное ограничение
tracking pixel, не исправляется на уровне CRM.

### Rate limiting

- 3 секунды между отправками (внутри кампании)
- Максимум 100 писем за один запуск кампании (`MAX_SENDS_PER_RUN`)
- Batch commit каждые 10 писем

## 5. Загрузка HTML-шаблонов в CRM

### Механизм

HTML-шаблоны загружаются **только через файл** — `<input type="file" accept=".html,.htm">`.
Редактирование HTML в textarea **не предусмотрено** — 150+ строк кода с inline-стилями
неудобно редактировать в текстовом поле.

Поток:
1. Пользователь выбирает `.html` файл на компьютере
2. Браузер читает файл через `FileReader.readAsText(file)`
3. Содержимое отправляется в API как `body` (string)
4. API сохраняет в `CrmTemplateRow.body`

### Обновление шаблона

Для обновления HTML-шаблона — повторная загрузка файла через тот же UI
(PUT /api/v1/templates/{name} с новым `body`).

### Лимит размера

`body` ограничен до 500 000 символов (~500KB). Этого достаточно для любого
email-шаблона с inline-стилями. Шаблоны с base64-изображениями превышают этот лимит
и не поддерживаются — используйте внешние URL для изображений.

## 6. Архитектура: будущее расширение

### Блочные шаблоны (не реализовано, задокументировано)

Текущая архитектура: один шаблон = один HTML-файл. В будущем возможен переход
к блочной системе:

```
Шаблон обёртки (header + footer + стили)
  + Контент-блок 1 (оффер «Ретушь для гравировки»)
  + Контент-блок 2 (оффер «Дизайн мемориальных комплексов»)
```

Для этого потребуется:
- Новая таблица `crm_template_blocks` (template_name, block_name, body, position)
- Рендеринг: собрать blocks → вставить в wrapper → отправить
- UI: drag-and-drop блоков

Текущая модель `body_type` (plain/html) совместима с этим расширением —
достаточно добавить уровень абстракции над `render()`.

### Встраивание изображений через CID (не реализовано)

Вместо внешних `<img src="https://...">` — внедрять изображения как MIME-вложения:

```python
from email.mime.image import MIMEImage

with open("photo.jpg", "rb") as f:
    img = MIMEImage(f.read())
    img.add_header("Content-ID", "<example_photo>")
    msg.attach(img)
```

В HTML: `<img src="cid:example_photo">`. Это обходит блокировку внешних
изображений в Gmail/Outlook. Требует хранения файлов изображений на сервере.

## 7. Известные проблемы и ограничения

| # | Проблема | Статус | Влияние |
|---|----------|--------|---------|
| 1 | Внешние изображения блокируются Gmail/Outlook | Задокументировано | Пользователь видит «Показать картинки», пример работ не виден сразу |
| 2 | Хостинг изображений (Netlify) может быть недоступен | Задокументировано | Картинка не загрузится, письмо покажет пустой блок |
| 3 | Tracking pixel — ложные срабатывания от прокси Gmail | Известное ограничение | `opened_at` может фиксироваться без реального прочтения |
| 4 | Plain-text альтернатива генерируется автоматически (strip tags) | By design | Может терять форматирование для сложных писем |
| 5 | HTML-шаблоны без плейсхолдеров = одинаковые письма всем | Текущее состояние | Нет персонализации, планируется добавить позже |
| 6 | follow-up маппинг (стадия → шаблон) захардкожен в коде | Известное ограничение | Нельзя настроить через UI, только изменение кода |

## 8. Чеклист перед запуском рассылки

- [ ] HTML-шаблон не содержит `<div style="display: none"...>` (plain-text блок)
- [ ] Все стили inline (в `style=""` атрибутах)
- [ ] Лейаут на `<table>`, не на `<div>`/flex/grid
- [ ] Присутствует `</body>` тег
- [ ] Внешние изображения доступны (URL не протух)
- [ ] Плейсхолдеры (если есть) — только в тексте тегов, не в атрибутах
- [ ] Тестовая отправка на свой email прошла успешно
- [ ] Контакты в футере корректны (Telegram, WhatsApp, имя)
