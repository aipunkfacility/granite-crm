## Часть 1. Основные проблемы предыдущих планов (до v4)

1. **Отсутствие воронки после ответа.** Планы заканчивались на «получили ответ». Переход `replied → interested → client` был не продуман.
2. **Follow-up через Telegram.** TG для рассылок массово заблокирован, follow-up должен быть только по email.
3. **Бессмысленные A/B тесты на 22 компаниях.** Выбирать победителя нужно по результатам отправки хотя бы 50 писем, а не 20. Смешивание аудитории и тем ломает тест.
4. **Ошибки в задержках и "зависаниях" системы.** Задержка писем внутри SSE (Server-Sent Events) соединения приведет к таймаутам балансировщика, и отправка прервется.
5. **Временные ссылки для отписки.** ngrok для продакшен-писем использовать нельзя — после закрытия терминала ссылки «Отписаться» перестают работать, что ведет к бану от спам-фильтров.
6. **Отсутствие отмены follow-up.** Задачи ставились автоматически с отложенной датой, но не отменялись, если клиент сам ответил раньше срока.

Решение: Этот план v5 решает все перечисленные архитектурные и логические ловушки.

## Часть 4. Технический план реализации

### Блокер 0: Архитектура отправки и Глобальный лимит

**Проблема:** SSE соединение будет разорвано сервером (таймаут 1-5 минут), если мы будем спать 120 минут на 50 адресах. Перезапуск кампании повторно спамит тех же получателей.
**Решение:** Отправка должна происходить в фоновой задаче FastAPI, а SSE просто поллит логи из БД.

1. Файл `granite/api/campaigns.py`:
```python

EMAIL_DELAY_MAX = int(os.getenv("EMAIL_DELAY_MAX", "120"))
EMAIL_DAILY_LIMIT = int(os.getenv("EMAIL_DAILY_LIMIT", "50"))

def run_campaign_in_background(campaign_id: int):
    # Фоновая функция. Создает отдельную сессию БД, цикл по отправке.
    # Обязательно проверяет глобальный лимит перед каждой отправкой!
    # ...
    pass

@router.post("/campaigns/{campaign_id}/run")
async def run_campaign(campaign_id: int, background_tasks: BackgroundTasks, db: Session = Depends(get_db)):
    campaign = db.get(CrmEmailCampaignRow, campaign_id)
    # ... валидация и проверка
    campaign.status = "running"
    db.commit()
    
    background_tasks.add_task(run_campaign_in_background, campaign_id)
    return {"status": "started"}

# SSE эндпоинт (/progress) больше не занимается отправкой.
# Он просто достает 'SELECT count(*) FROM crm_email_logs WHERE campaign_id=X' 
# и возвращает это в EventSource раз в 3 секунды.
```

2. **Защита от превышения лимитов:**
Внутри цикла `run_campaign_in_background` перед каждым `sender.send()`:
```python
last_24h = datetime.now(timezone.utc) - timedelta(days=1)
sent_last_24h = (
    local_db.query(CrmEmailLogRow)
    .filter(CrmEmailLogRow.sent_at >= last_24h)
    .count()
)
if sent_last_24h >= EMAIL_DAILY_LIMIT:
    campaign.status = "paused_due_to_limit"
    local_db.commit()
    break  # Прерываем кампанию до завтра
```

### Блокер 1: Статичный URL через Cloudflare Tunnel

Ngrok запрещен для писем: ссылки отписки станут 404 при перезагрузке ПК, и вы получите бан.
Указывать в `.env` переменную `BASE_URL` можно только на статичный адрес.

```bash
cloudflared tunnel login
cloudflared tunnel create granite-crm
cloudflared tunnel route dns granite-crm crm.yourdomain.com
cloudflared tunnel run granite-crm
```
В `.env`: `BASE_URL=https://crm.yourdomain.com`

### Задача 3: Детерминированный A/B тест (Хеширование)

В `granite/database.py` добавляем столбцы `subject_a`, `subject_b` (String, nullable) в `CrmEmailCampaignRow` и генерируем миграцию Alembic.
В `sender.py` корректное распределение:
```python
import hashlib

def get_ab_variant(company_id: int) -> str:
    """Для малых выборок %2 может косить. Хеш распределяет равномерно и детерминированно."""
    hash_val = int(hashlib.md5(str(company_id).encode()).hexdigest(), 16)
    return "A" if hash_val % 2 == 0 else "B"
```


### Задача 4: Воронка ответов и Авто-отмена задач

Самое важное: не забыть отменить follow-up, если клиент уже вышел на связь!

При обновлении контакта вручную (`PATCH /api/v1/companies/{company_id}`), проверяем смену статуса:
```python
if new_data.funnel_stage in ["replied", "interested", "client"]:
    db.query(CrmTaskRow).filter(
        CrmTaskRow.company_id == contact.company_id,
        CrmTaskRow.status == "pending",
        CrmTaskRow.task_type == "follow_up"
    ).update({"status": "canceled"}, synchronize_session=False)
```

Автоматическое создание: в `tracking.py`, при открытии письма, если задачи еще нет: ставим через 3 дня.
