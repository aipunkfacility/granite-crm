# Аудит кодовой базы: GRANITE CRM

## 1. Резюме
GRANITE CRM — это легковесная локальная CRM-система, построенная на FastAPI (Python) и Vanilla JS. Архитектура проекта «local-first»: данные хранятся в IndexedDB (Dexie.js) для высокой скорости работы интерфейса и синхронизируются с JSON-файлами на сервере для надежного хранения и создания бэкапов.

### Используемые технологии
- **Backend:** Python (FastAPI, Pydantic, smtplib, uvicorn)
- **Frontend:** HTML5, Vanilla JS, CSS (Custom Variables), Dexie.js (IndexedDB), Remix Icon
- **Хранение:** JSON-файлы (Сервер) + IndexedDB (Клиент)
- **Связь:** SSE (Server-Sent Events) для прогресса рассылок, REST API для CRUD-операций

---

## 2. Сильные стороны
- **Производительность:** Использование IndexedDB для всех операций в UI обеспечивает мгновенный отклик, а JSON-файлы делают базу данных переносимой и читаемой человеком.
- **Система бэкапов:** Реализованы автоматические бэкапы (N версий) и атомарная запись файлов (через временные файлы и переименование), что гарантирует целостность данных.
- **Реалтайм интерфейс:** SSE в модуле рассылки писем реализован профессионально и дает пользователю наглядную обратную связь.
- **Безопасность:** Для локального приложения грамотно настроены CORS (только localhost), защита от Path Traversal и аутентификация по токену.

## 3. Проблемы и уязвимости

### 🔴 Критические / Высокий приоритет
- **Отсутствующий импорт:** В файле `server/routers/db.py` функция `restore_backup` использует `shutil.copy2`, но сам модуль `shutil` не импортирован. Это вызовет `NameError` при попытке восстановления из бэкапа.
- **Блокирующий I/O в async-цикле:** Библиотека `smtplib` является синхронной и блокирующей. Ее использование напрямую внутри `async` функций (в `server/services/email.py`) блокирует весь событийный цикл FastAPI на время установки соединения и отправки писем. При больших рассылках сервер может перестать отвечать на другие запросы (например, проверку статуса).

### 🟡 Средний приоритет
- **Ручное версионирование ассетов:** Использование `?v=10` в `index.html` делает процесс обновления кэша ручным и подверженным человеческим ошибкам.
- **«Хакерский» ввод токена:** В `server/__init__.py` реализована замена строки в `index.html` при каждой загрузке страницы для передачи API-токена. Это работает для локального приложения, но может сломаться при изменении шаблона.

### 🟢 Низкий приоритет / Предложения по улучшению
- **Обработка ошибок:** В `js/db.js` функции `loadFromServer` и `saveToServer` имеют только базовый `try/catch`. Стоит добавить логику повторных попыток или более детальную индикацию статуса в UI.
- **Дублирование логики:** Логика именования файлов бэкапов дублируется в `server/routers/db.py` и `server/services/backup.py`.

## 4. Рекомендованный план действий

### Срочные исправления (Баги)
1. **Добавить импорт `shutil`** в `server/routers/db.py`.
2. **Перейти на `aiosmtplib`** или выполнять SMTP-операции в отдельном потоке (через `run_in_executor`), чтобы не блокировать сервер.

### Улучшение архитектуры
1. **Рефакторинг передачи токена:** Вместо модификации HTML на лету сделать отдельный эндпоинт `/config` или передавать токен через стандартные заголовки после первого входа.
2. **Централизация работы с JSON:** Перенести всю логику сопоставления «Область -> Файл» на бэкенд, чтобы избежать дублирования логики в JS и Python.
