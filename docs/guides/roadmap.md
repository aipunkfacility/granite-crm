# Granite CRM — Дорожная карта для одного разработчика

> Формат: практический план без воды  
> Аудитория: вайбкодер, один пользователь, локально  
> Цель: живой рабочий продукт, а не идеальный код  
> Дата: апрель 2026

---

## Главный принцип

**Работающее плохо > Идеальное никогда.**

Ты уже неделю исправляешь баги, которые не мешают тебе работать. Пора остановиться и начать использовать то, что уже есть. Фронтенд — это то, что даст тебе реальную обратную связь о том, что действительно нужно фиксить.

## Фаза 1: Фронтенд MVP ✅ (завершено)

### Общий подход

Фронтенд — это не отдельный проект, это интерфейс к тому, что уже работает. Начинать с самого простого экрана и добавлять по одному.

### Технологический стек (рекомендации)

**Рекомендую: Next.js + Tailwind CSS + shadcn/ui**

Почему:
- shadcn/ui — готовые компоненты для таблиц, карточек, форм. Не нужно верстать с нуля
- Next.js — можно деплоить позже без переписывания
- TanStack Query — удобный слой для запросов к API, кэширование из коробки
- TypeScript — поможет не ошибиться в именах полей API

Альтернатива для быстрого старта: **Vite + React + Tailwind** — чуть проще в настройке.

**Не рекомендую:**
- Vue, Svelte — меньше готовых CRM-компонентов
- Vanilla JS — слишком много ручной работы для таблиц и форм
- Angular — избыточно для одного пользователя

### Экраны в порядке приоритета

#### Экран 1: Список компаний (сделать первым)

Это главная страница. На ней должны быть:
- Таблица с колонками: название, город, сегмент (A/B/C/D), телефон, Telegram, CRM-статус
- Цветовые метки сегментов (A — зелёный, D — серый)
- Поиск по названию
- Фильтры: город, сегмент, есть Telegram, воронка
- Пагинация (50 на страницу)
- Клик по строке — открывает карточку компании

Это всё уже есть в API (`GET /companies` с параметрами). Просто нужно отрисовать.

#### Экран 2: Карточка компании

Открывается по клику из списка. Содержит:
- Контакты: телефоны, email, сайт, мессенджеры
- CRM-статус и кнопка смены стадии воронки
- Поле для заметок
- История касаний (внизу страницы)
- Кнопки: отправить в TG, отправить в WA (пока просто копирует ссылку)

#### Экран 3: Follow-up очередь

Страница "Кому писать сегодня". API уже есть (`GET /followup`). Показывает:
- Список компаний, которым пора написать
- Рекомендованный канал (email/TG/WA)
- Кнопка "Написал" — меняет стадию воронки

Это самый полезный экран для реальной работы.

#### Экран 4: Задачи

Простой список задач с фильтром по статусу. API есть. Приоритет — ниже, чем первые три.

#### Экран 5: Статистика

Дашборд с цифрами: всего компаний, по сегментам, с Telegram, с email. API есть (`GET /stats`). Можно сделать позже, когда остальное работает.

### Что НЕ делать во фронтенде (пока)

- Не делать страницу запуска пайплайна из UI — запускай через CLI
- Не делать отправку email через UI — это работает, но сложно тестировать
- Не делать WebSocket/real-time обновления — поллинг каждые 30 секунд вполне достаточно
- Не делать мобильную версию — один пользователь на ноутбуке
- Не делать красивые графики и диаграммы — таблицы достаточно

### Рабочий процесс с API

API запускается командой `uv run cli.py api --port 8000`. Фронтенд делает запросы на `http://localhost:8000/api/v1/`. CORS уже настроен для `localhost:3000` и `localhost:5173`.

Документация API доступна по адресу `http://localhost:8000/docs` — там можно тестировать каждый эндпоинт прямо в браузере. Использовать это активно при разработке фронтенда.

## Фаза 2: Стабилизация (частично завершено — email-кампании работают)

### Частично завершено

Фронтенд работает и используется ежедневно. Email-кампании с SSE-прогрессом, tracking pixel, A/B тестирование тем — всё реализовано. Дальнейшая стабилизация — по мере реального использования.

### Что включать в эту фазу

**Soft-delete везде** — если при работе с фронтендом замечаешь, что удалённые компании появляются где-то ещё.

**Исправление scoring** — если сегменты не соответствуют ожидаемому (много A-компаний без данных или много D-компаний с данными).

**Улучшение фильтров** — добавить те фильтры, которых не хватает при реальной работе.

**Починка enrichment-ошибок** — только тех, что видишь в логах при реальном использовании.

### Что откладывать дальше

**Все аудиты 2–10 из матрицы GLM** — они описывают проблемы production-системы с тысячами пользователей. У тебя один пользователь, локально. SSRF, timing-attack, DNS rebinding — это не твои проблемы прямо сейчас.

**WebSocket и real-time** — поллинг каждые 30 секунд для одного пользователя более чем достаточно.

**Redis, multi-worker, gunicorn** — SQLite + один uvicorn-воркер отлично работает для одного пользователя.

**Dockerfile и production deployment** — когда появятся клиенты.

**Reverse lookup (2GIS/Yell enrichment)** — он отключён в конфиге по причине. Не включать до тех пор, пока основной поток не работает стабильно.

## Фаза 3: Монетизация (когда есть клиенты)

### Что будет нужно при переходе к multi-user

Это другой продукт. Переход от "инструмент для себя" к "сервис для клиентов" требует переосмысления архитектуры, а не просто добавления фич.

**Минимум для первого клиента:**
- Аутентификация пользователей (сейчас её нет совсем)
- PostgreSQL вместо SQLite (или хотя бы перенос на отдельный сервер)
- Базовый мониторинг (что система вообще работает)
- Резервное копирование по расписанию

**Что менять в API при переходе к multi-user:**
- Добавить `user_id` или `tenant_id` к основным таблицам
- Изоляция данных между клиентами
- Rate limiting уже есть в коде — просто настроить

**Когда думать об этом:** когда есть первый платящий клиент, а не раньше.

## CRM: что работает хорошо и что не трогать

### Что уже хорошо реализовано

- Воронка с 9 стадиями — достаточно для аутрича
- Follow-up логика (когда и каким каналом писать) — умная и рабочая
- Tracking-pixel для email открытий — редкая фича для такого уровня проекта
- Scoring A/B/C/D — понятная сегментация

### Что стоит упростить

**Email-кампании** — это сложная фича. Если реально нужна массовая рассылка, использовать внешний сервис (SendGrid, Unisender). Собственный SMTP-отправщик — это постоянная боль с доставляемостью, черными списками, SPF/DKIM. Кампании в коде оставить как есть, но не развивать.

**Шаблоны** — текущая система с `{placeholder}` простая и работает. Не усложнять до Jinja2 или MDX.

**Задачи** — хватит текущего минимума (создать, закрыть, фильтровать). Не делать Kanban, не делать назначение, не делать дедлайны с уведомлениями.

## Пайплайн: что не трогать

### Правило пайплайна

Пайплайн работает. Он медленный, иногда ошибается, иногда находит что не то — но он работает. Не трогать до тех пор, пока нет конкретной жалобы из реального использования.

### Что точно не делать

- Не включать async_enabled — синхронный режим стабильнее, пусть медленнее
- Не включать dgis и yell по умолчанию — они работают, но требуют Playwright и капризны
- Не включать reverse_lookup — дополнительные запросы к 2GIS за компании без данных
- Не переписывать dedup — Union-Find работает, fuzzy matching работает

### Как работать с пайплайном

Запускать руками когда нужно: `python cli.py run "Город"`. Смотреть логи в `data/logs/granite.log`. Если упал — смотреть на каком этапе (scraping/dedup/enrichment) и перезапускать с нужным флагом.

Для ежедневной работы пайплайн не нужен вообще — данные уже есть в базе.

## Архитектурные решения: что оставить как есть

### SQLite — оставить

SQLite с WAL-режимом отлично работает для одного пользователя с базой в 6000–50000 записей. Быстро, просто, без настройки. Мигрировать на PostgreSQL только когда действительно нужна параллельная запись от нескольких пользователей.

### Один процесс — оставить

Один uvicorn без workers для локального использования — это правильно. Меньше сложности.

### uv как менеджер пакетов — оставить

Это хорошее решение. Не менять на pip или poetry.

### Alembic миграции — оставить, но не добавлять новые без необходимости

Схема базы достаточно зрелая. Новые миграции только если реально нужно новое поле. Не делать миграции "для порядка".

## Расставление приоритетов: простая матрица

Когда появляется идея или баг — задавать себе три вопроса:

**1. Это мешает работать прямо сейчас?**
- Да → исправить
- Нет → записать, вернуться позже

**2. Это видно в UI?**
- Да → высокий приоритет
- Нет → низкий приоритет

**3. Это нужно для клиентов?**
- Нет клиентов → отложить
- Есть клиент → обсудить с клиентом

Большинство аудитных замечаний не проходят ни один из этих фильтров.

Аудиты написаны правильно и профессионально. Проблема в том, что они написаны для production-системы с несколькими пользователями и под угрозой атак.

**Аудит 1 (Безопасность API)** — актуален только когда система публично доступна. Локально — нет смысла.

**Аудит 2 (SSRF)** — is_safe_url() уже реализован. DNS rebinding на локальной машине — теоретическая угроза.

**Аудит 3 (Схема БД)** — часть уже исправлена (миграции есть). Остальное — для production.

**Аудиты 4–10** — хорошие идеи для версии 2.0. Не для сейчас.

**Рекомендация:** сохранить файл аудитов. Открыть когда появится первый клиент.

## Чеклист готовности к фронтенду

Перед тем как начать писать фронтенд, убедиться:

- [ ] `python cli.py api --port 8000` запускается без ошибок
- [ ] `http://localhost:8000/health` возвращает `{"status": "ok", "db": true}`
- [ ] `http://localhost:8000/docs` открывается и показывает все эндпоинты
- [ ] `GET /api/v1/companies` возвращает список компаний
- [ ] `GET /api/v1/companies?city=Москва` фильтрует по городу
- [ ] `GET /api/v1/stats` возвращает статистику
- [ ] soft-delete фикс применён (удалённые компании не видны)
- [ ] Бэкап базы сделан

Если всё это работает — можно начинать фронтенд. Бэкенд достаточно хорош.
